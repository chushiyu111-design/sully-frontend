# Low-load Verification Policy

本仓库的日常开发机器性能有限。验证目标不是机械地跑完所有命令，而是在不拖垮本机的前提下，把本次改动的主要风险说清楚、查到位。

## 默认原则

- 默认不运行全量测试、全量构建、全量 E2E、coverage、benchmark、长时间 watch。
- 优先验证本次修改直接影响的文件、组件、页面和调用链。
- 优先使用代码阅读、最小范围命令、浏览器冒烟、控制台检查来覆盖风险。
- 需要重型验证时，先说明原因和可能耗时；除非用户明确要求，不主动执行。
- 最终回复必须说明实际做过的验证，以及因为性能或范围原因没有做的验证。

## 验证分层

### 轻量验证，默认优先

- 阅读相关代码路径，确认调用链和状态流没有断裂。
- 针对修改文件做静态检查或最小范围搜索。
- 运行单个相关测试文件，或最小相关测试集合。
- 页面改动优先复用已有 dev server；必要时启动本地页面并做关键路径冒烟。
- 检查浏览器控制台是否出现明显错误。

### 中等验证，按需执行

- 运行相关模块测试，而不是全仓测试。
- 运行耗时可控的 `tsc`、lint 或构建子集。
- 对核心交互做手动冒烟，例如打开页面、点击关键按钮、检查响应式布局。
- 修改共享组件、公共 hook、运行时配置时，可以扩大到相邻功能验证。

### 重型验证，默认不执行

- `npm run test:run` 等全量测试。
- 全量 `npm run build`。
- 全量 Playwright / E2E。
- coverage、benchmark、profiling。
- 长时间运行的 watch、批处理、批量生成任务。

重型验证只在以下情况考虑执行：

- 用户明确要求。
- 发布前必须确认，且用户知道它可能明显占用机器。
- 修改触达核心入口、数据库迁移、部署脚本、全局运行时配置等高风险区域。
- 轻量和中等验证无法合理覆盖本次风险。

## 可以跳过全量测试的情况

以下情况默认不跑全量测试：

- 只修改文档、注释、说明性文本。
- 只改局部样式、布局、文案，且影响范围清楚。
- 修改单个组件或单条调用链，可以用代码阅读和局部冒烟覆盖。
- 当前机器资源不足，运行全量命令可能造成卡顿、卡死或严重拖慢。

跳过不等于不验证。需要选择合适的轻量替代方式，并在最终回复中写清楚。

## 推荐替代方式

- 说明已阅读的关键文件和调用链。
- 运行最小相关测试，例如指定单个 `*.test.tsx`。
- 运行最小相关类型检查或搜索命令。
- 用浏览器打开受影响页面，确认无明显渲染异常和控制台错误。
- 给出人工验收点，方便用户在真实环境里快速点一遍。

## 回复格式要求

完成任务时，按实际情况简短说明：

- 已验证：列出实际运行或检查过的内容。
- 未运行：列出跳过的重型命令和原因。
- 风险提示：只有存在残余风险时才写。

示例：

```text
已验证：阅读了设置页调用链，并运行了 apps/settings/ApiSettings.test.tsx。
未运行：未跑全量 test/build，按仓库低负载策略跳过。
```

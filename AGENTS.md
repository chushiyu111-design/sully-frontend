# Repo Agent Rules

本仓库是前端仓库，负责页面、组件、样式、浏览器侧逻辑与前端代理调用。

## 进入本仓库前先遵守

- 优先遵守上级工作区的编码规则；如果你是单独打开本仓库，也必须执行下面的字符编码规则
- 所有新增或修改的文本文件默认使用 UTF-8
- `*.ps1`、`*.psm1`、`*.psd1` 如果包含非 ASCII 字符，使用 UTF-8 with BOM
- 文本读写显式指定 UTF-8，不依赖 Windows 默认编码
- 修改中文内容后，重新打开文件确认没有乱码
- 如果只是在终端里看到乱码，先判断终端编码，不要立刻覆盖文件

## 仓库范围

- React、Vite、Tailwind、页面结构、前端配置、前端调用链路，都在本仓库内处理

## 与用户沟通方式

- 用户没有计算机基础，沟通时先用自然语言解释目标、影响范围和风险，再进入技术细节。
- 不要急于直接给完整方案或改代码；在需求不清楚时，先复述理解并用少量问题引导用户确认。
- 改动功能前，主动判断可能被误伤的相关功能、共享组件、状态、接口和样式，并提醒用户。
- 优先给用户可理解的选项，例如“保守修复 / 彻底重构 / 先做测试版”，并说明推荐理由。
- 修改完成后，用“改了什么、怎么验收、可能还有什么风险”的格式汇报。
- 涉及 `main`、正式部署、Cloudflare production 时，必须明确提醒风险并等待用户明确授权。

## 新手协作保护

- 用户是非计算机专业背景，Codex 应主动降低理解成本，用自然语言解释每一步的目的、影响和风险。
- 默认采用小步、安全、可回退的改法；避免为了局部问题做大范围重构。
- 涉及删除文件、大范围改动、登录、支付、数据、部署、环境变量、依赖升级等高风险操作时，必须先暂停并向用户确认。
- 修改前说明可能被影响的页面、组件、状态、接口或样式；修改后检查核心关联功能是否被误伤。
- 修改完成后必须提供清晰验收步骤，让用户知道打开哪里、点击哪里、看到什么代表成功。
- 遇到不确定的代码、配置或部署行为时，不要猜测执行；先查证，并用自然语言说明不确定点。
- 不要覆盖、回退或删除用户已有改动；发现非本次产生的改动时，应保留并提醒用户。

## 产品定位与审美

- 本项目是免费开源的女性向 AI RP / 陪伴类产品，前端体验应服务于角色沉浸、情绪陪伴、长期关系感和用户安全感。
- 文案要温柔、细腻、克制、有边界感，避免技术腔、命令感、爹味说教、油腻营销或过度夸张的表达。
- UI 要符合女性向审美，优先高级、干净、柔和、精致、易读的体验；不要为了炫技牺牲可用性、可读性和沉浸感。
- 新增页面、弹窗、空状态、按钮、提示语时，要主动检查是否符合本项目气质，而不是套用通用 SaaS、后台管理或硬核工具风格。
- 因为项目免费开源，涉及收费、商业化、埋点追踪、账号体系、数据上传等设计时，必须先提醒用户潜在影响并等待明确确认。

## 部署红线

- 本仓库的 `origin/main` 会自动触发 Cloudflare Pages 正式环境部署；推送到 `main` 就等于更新正式链接。
- 未经用户在当前对话中明确授权“更新正式链接”“发布生产”“推送 main”，禁止执行：
  - `git push origin main`
  - 合并分支到 `main`
  - 任何 Cloudflare Pages production 部署
- 用户提到“测试链接”“beta”“预览”“测试环境”时，只允许使用测试发布路径，不允许以任何形式碰 `main`。
- 前端测试发布默认只使用 `./deploy-beta.ps1`。
- 如果需要临时验证代码、保留提交或隔离变更，优先使用临时分支或 worktree；不要把 `main` 当测试发布通道。
- 在执行任何推送前，先确认目标分支是否会自动触发正式环境；只要不确定，就视为 production 风险并停止。
